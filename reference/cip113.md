# CIP-113: Programmable Token Logic

Reference for implementing CIP-113 programmable tokens in Aiken with MeshJS
off-chain. Based on the [Cardano Foundation specification](https://cips.cardano.org/cip/CIP-0113)
and validated end-to-end on preview testnet via
[adavault/programmable-tokens](https://github.com/adavault/programmable-tokens).

## Overview

CIP-113 defines a multi-validator architecture for tokens with enforced
transfer rules. Unlike native tokens (which anyone can freely move), programmable
tokens must pass through a validator pipeline on every transfer — enabling
whitelists, blacklists, transfer limits, freeze/seize, and compliance controls.

**Key insight:** The spending validator runs per-input (expensive with many UTxOs).
CIP-113 keeps the spend validator O(1) by delegating all logic to a single
withdraw-zero coordinator that runs once per transaction.

## Architecture

```
                    ┌─────────────────────────────────┐
                    │   programmable_logic_global      │
                    │   (withdraw — runs ONCE)         │
                    │                                  │
                    │   1. Sum authorized inputs       │
                    │   2. Check registry proofs       │
                    │   3. Invoke transfer logic       │
                    │   4. Verify output preservation  │
                    └──────────┬───────────────────────┘
                               │
              ┌────────────────┼────────────────┐
              ▼                ▼                 ▼
┌──────────────────┐ ┌─────────────────┐ ┌──────────────┐
│ prog_logic_base  │ │ transfer_logic  │ │   registry   │
│ (spend — O(1))   │ │ (withdraw)      │ │ (mint+spend) │
│                  │ │                 │ │              │
│ Checks global    │ │ Token-specific  │ │ Sorted       │
│ is in withdrawals│ │ rules: whitelist│ │ linked list  │
│                  │ │ limits, KYC     │ │ of policies  │
└──────────────────┘ └─────────────────┘ └──────────────┘
```

### Validators (12 total in reference implementation)

| Validator | Purpose | Handler | Runs per |
|-----------|---------|---------|----------|
| `programmable_logic_base` | Custody all programmable UTxOs | spend | input |
| `programmable_logic_global` | Transaction-level coordinator | withdraw | tx |
| `protocol_params_mint` | One-shot mint of params NFT | mint | tx |
| `registry_mint` | Registry linked list operations | mint | tx |
| `registry_spend` | Guard registry node UTxOs | spend | input |
| `issuance_cbor_hex_mint` | One-shot mint of CBOR hex template | mint | tx |
| `issuance_mint` | Per-token minting policy | mint | tx |
| `transfer_logic` | Open transfer (admin sig only) | withdraw | tx |
| `loyalty_transfer_logic` | Whitelist-based transfer | withdraw | tx |
| `loyalty_minting_logic` | Admin-only mint/burn | withdraw | tx |
| `third_party_transfer_logic` | Seize/freeze admin actions | withdraw | tx |

## Address Model

All programmable tokens live at addresses with the **same payment credential**
(the `programmable_logic_base` script hash) but **different stake credentials**
(per-holder VK or script hash).

```
prog_logic_address = Script(prog_logic_hash) + Inline(VK(holder_key_hash))
```

This means:
- **Ownership** is determined by the stake credential, not the payment credential
- **Authorization** requires the stake credential holder to sign (VK) or be
  invoked (script) — checked by the global coordinator
- **All holders share one spend validator** — the global coordinator processes
  all tokens in one pass regardless of how many holders are involved

### Building the Address (MeshJS)

```typescript
import { buildBaseAddress, CredentialType } from "@meshsdk/core-cst";

function buildProgLogicAddress(
  progLogicHash: string,    // script hash
  holderKeyHash: string,    // VK hash of the token holder
  networkId: number,        // 0 = testnet, 1 = mainnet
): string {
  const addr = buildBaseAddress(
    networkId,
    progLogicHash,
    holderKeyHash,
    CredentialType.ScriptHash,  // payment = script
    CredentialType.KeyHash,     // stake = VK
  );
  return addr.toAddress().toBech32();
}
```

## Registry Design

The registry is a **sorted linked list** of registered programmable token
policies, stored on-chain as UTxOs with NFT markers.

### Registry Node Structure

```aiken
type RegistryNode {
  key: ByteArray,                              // policy ID of registered token
  next: ByteArray,                             // next key (lexicographic order)
  transfer_logic_script: Credential,           // withdraw validator for transfers
  third_party_transfer_logic_script: Credential, // withdraw validator for admin ops
  global_state_cs: ByteArray,                  // optional state NFT (e.g., denylist)
}
```

### Origin Node

The registry starts with an origin node where `key = ""` and
`next = 0xff...ff`. All registered policies are inserted between these bounds.

### Membership Proofs

The global coordinator uses two proof types per non-ADA policy in the transaction:

- **`TokenExists { node_idx }`** — Reference input at `node_idx` has
  `node.key == policy_id`. The transfer logic MUST be in withdrawals.
- **`TokenDoesNotExist { node_idx }`** — Covering node proves
  `node.key < policy_id < node.next` (non-membership). Token is skipped.

## Protocol Parameters

A one-shot NFT (`ProtocolParams`) stores on-chain configuration:

```aiken
type ProgrammableLogicGlobalParams {
  registry_node_cs: PolicyId,   // policy ID of registry NFTs
  prog_logic_cred: Credential,  // payment credential of programmable_logic_base
}
```

Referenced as a read-only input in every programmable token transaction.

## Transaction Flows

### Bootstrap (Phase 1)

Establish the protocol infrastructure. Four transactions:

1. **Mint protocol params NFT** — One-shot mint, inline datum with registry CS
   and prog logic credential
2. **Initialize registry** — Mint origin node NFT with `RegistryInit` redeemer,
   create UTxO with `RegistryNode { key: "", next: 0xff...ff, ... }` datum
3. **Mint issuance CBOR template** — One-shot mint storing the parameterized
   minting policy prefix/postfix for verification during registration
4. **Register stake credentials** — `registerStakeCertificate()` for global,
   transfer, and minting logic validators (Conway: permissionless, no witness)

### Register Token (Phase 2)

Insert a new programmable token policy into the registry:

1. Find the covering node (where `node.key < new_policy < node.next`)
2. Mint a new registry NFT with `RegistryInsert` redeemer
3. Create new node: `{ key: new_policy, next: covering.next, ... }`
4. Update covering node: `{ ...covering, next: new_policy }`

### Deploy Reference Scripts (Phase 2.5)

Store validator scripts on-chain as reference UTxOs to reduce transaction size:

```typescript
// Each output carries one script — Kupo indexes by credential
.txOut(walletAddress, [{ unit: "lovelace", quantity: "30000000" }])
.txOutReferenceScript(globalScriptCbor, "V3")
```

### Mint Tokens (Phase 3)

Four validators in one transaction:

```
issuance_mint (mint)              — checks global + minting logic invoked
programmable_logic_global (withdraw) — validates registry proof, output preservation
loyalty_minting_logic (withdraw)     — checks admin signature
transfer_logic (withdraw)            — in withdrawals for value check pass-through
```

**Redeemer for issuance_mint:**
```typescript
{ constructor: 0, fields: [
  { constructor: 1, fields: [{ bytes: mintingLogicHash }] }  // Script credential
]}
```

**Redeemer for global (TransferAct):**
```typescript
{ constructor: 0, fields: [
  { list: [] },                                               // proofs (no inputs)
  { list: [{ constructor: 0, fields: [{ int: nodeIdx }] }] }  // mint_proofs
]}
```

**Execution budget:** With 4 validators, the Ogmios evaluator conservatively
estimates ~21M memory (7M default per redeemer). Actual usage is ~15M. Use
manual exUnits to stay within the 16.5M protocol limit:

```typescript
const exBudget = {
  issuanceMint:     { mem: 3_000_000, steps: 1_500_000_000 },
  globalWithdraw:   { mem: 8_000_000, steps: 4_000_000_000 },
  mintingWithdraw:  { mem: 2_000_000, steps: 1_000_000_000 },
  transferWithdraw: { mem: 2_000_000, steps: 1_000_000_000 },
  // Total:          15M mem,          7.5B steps
};
const txBuilder = new MeshTxBuilder({
  fetcher: kupo,
  submitter: ogmios,
  // No evaluator — using explicit exUnits
});
```

### Transfer Tokens (Phase 4)

Three validators in one transaction:

```
programmable_logic_base (spend)      — O(1) global withdrawal check
programmable_logic_global (withdraw) — sum inputs, check proofs, verify outputs
transfer_logic (withdraw)            — token-specific rules
```

**Authorization model:** The global coordinator checks each input's stake
credential:
- `VerificationKey(pkh)` → must be in `extra_signatories`
- `Script(hash)` → must be in `withdrawals`

This means a transfer from holder A to holder B requires **A's signature**
(as the stake VK holder of the input). The admin must also sign if the
`transfer_logic` requires it.

**Multi-holder transfer (round-trip example):**
```typescript
// Spend from both sender and recipient addresses in one tx
.spendingPlutusScriptV3()
.txIn(recipientUtxo...)  // recipient's stake VK must sign
.spendingPlutusScriptV3()
.txIn(senderUtxo...)     // sender's stake VK must sign
// Both required signers
.requiredSignerHash(senderKeyHash)
.requiredSignerHash(recipientKeyHash)
// Sign with both keys
.signingKey(senderSigningKey)
.signingKey(recipientSigningKey)
```

### Third-Party Actions (Seize/Freeze)

Uses the `ThirdPartyAct` redeemer on the global coordinator. The
`third_party_transfer_logic_script` must be in withdrawals (not the regular
transfer logic).

**Constraints:**
- Input/output pairs must preserve address and datum
- Non-policy assets must be unchanged per pair
- Policy tokens must actually change (anti-DDOS)
- Total output tokens >= total input tokens (no destruction)

## Withdraw-Zero Pattern

All transaction-level validators are invoked via withdraw-zero: a withdrawal
of 0 ADA from a script stake credential. This is the core coordination
mechanism:

```typescript
// Register stake credential first (one-time, permissionless in Conway)
txBuilder.registerStakeCertificate(rewardAddress);

// Then invoke via withdrawal in every tx
txBuilder
  .withdrawalPlutusScriptV3()
  .withdrawal(rewardAddress, "0")         // 0 ADA withdrawal
  .withdrawalScript(scriptCbor)           // inline script
  .withdrawalRedeemerValue(redeemer, "JSON", exUnits);
```

**Reward address construction:**
```typescript
import { serializeRewardAddress } from "@meshsdk/core";

const rewardAddr = serializeRewardAddress(
  scriptHash,    // 28-byte script hash
  true,          // isScript = true
  networkId,     // 0 = testnet
);
```

## UTxO Management

### Exclusion Strategy

Programmable token transactions use many UTxOs as reference inputs. These
must NOT be consumed by the input selector:

```typescript
const excludedIds = new Set([
  `${ppUtxo.input.txHash}#${ppUtxo.input.outputIndex}`,       // params NFT
  `${registryNodeUtxo.input.txHash}#${registryNodeUtxo.input.outputIndex}`, // registry
]);
// Reference script UTxOs (if deployed)
if (state.refScriptsTxHash) {
  for (let i = 0; i < 5; i++) {
    excludedIds.add(`${state.refScriptsTxHash}#${i}`);
  }
}
const selectableUtxos = walletUtxos.filter(
  (u) => !excludedIds.has(`${u.input.txHash}#${u.input.outputIndex}`)
);
txBuilder.selectUtxosFrom(selectableUtxos);
```

### Script UTxO Tracking

Programmable token UTxOs at script addresses move with each transfer. Track
the latest location:

```typescript
// LOYAL UTxO moves: mintTxHash#0 → transferTxHash#0 → ...
const loyaltyTxHash = state.transferTxHash || state.mintTxHash;
const loyaltyUtxo = {
  txHash: loyaltyTxHash,
  outputIndex: 0,
  amount: [...],
  address: progLogicAddr,
};
// Must provide all 5 .txIn() params + scriptSize=0 for offline UTxOs
.txIn(loyaltyUtxo.txHash, loyaltyUtxo.outputIndex,
      loyaltyUtxo.amount, loyaltyUtxo.address, 0)
```

### Reference Input Sorting

Cardano sorts reference inputs lexicographically by `(txHash, outputIndex)`.
Registry proof indices (`node_idx`) must match this sorted order:

```typescript
const refInputsSorted = [
  { hash: ppUtxo.input.txHash, idx: ppUtxo.input.outputIndex, type: "pp" },
  { hash: registryUtxo.input.txHash, idx: registryUtxo.input.outputIndex, type: "registry" },
].sort((a, b) => a.hash < b.hash ? -1 : a.hash > b.hash ? 1 : a.idx - b.idx);

const registryNodeIdx = refInputsSorted.findIndex((r) => r.type === "registry");
```

## Security Considerations

### Value Preservation

The global coordinator enforces: `output_prog_value >= input_prog_value + mint_value`.
This prevents tokens from "escaping" the programmable address system.

### Registry Proof Exhaustion

Every non-ADA policy in the inputs must have a corresponding proof. The
validator checks `remaining_proofs == []` after processing all policies.
This prevents unregistered tokens from being silently transferred.

### `expect` vs Return

Use `expect has_key(tx.withdrawals, ...)` not bare `has_key(...)` as a return
value. If `has_key` returns `False` for a `TokenExists` proof, the token would
be silently treated as non-programmable — bypassing transfer rules entirely.

### Stake Credential Authorization

The global coordinator enforces that each input's stake credential holder
authorized the spend. For VK credentials, the holder must be in
`extra_signatories`. For script credentials, the script must be invoked via
withdrawal. This prevents unauthorized spending of programmable tokens.

### Third-Party Anti-DDOS

The `ThirdPartyAct` redeemer requires that the seized policy's token quantities
actually change between input and output pairs. This prevents no-op
transactions that would waste on-chain resources.

## E2E Testing Approach

The reference implementation uses a phased E2E test harness with persistent
state across phases:

```
Phase 1: bootstrap    → params NFT + registry + issuance template + stake regs
Phase 2: register     → insert LOYAL token into registry
Phase 3: deploy       → reference scripts on-chain (optional optimization)
Phase 4: mint         → 1000 LOYAL tokens at prog logic address
Phase 5: transfer     → round-trip between different holders
```

**State persistence:** Each phase writes outputs (tx hashes, policy IDs, script
CBOR) to a JSON state file. Subsequent phases load this state to build on
previous results.

**Infrastructure required:** Ogmios (tx eval + submit) + Kupo (UTxO indexer)
connected to a preview/preprod node. SSH tunnels if node is remote.

**Kupo patterns:** Register script credentials for all deployed contracts:
```bash
# Wallet credential
--match "0027a6fe.../*"
# Programmable logic base script
--match "064b53ad.../*"
# Plus all other script credentials from deployed validators
```

## Execution Budget Reference

Protocol limits: **16.5M memory / 10B steps** per transaction.

| Transaction | Validators | Actual Memory | Evaluator Estimate |
|-------------|-----------|---------------|-------------------|
| Mint (4 scripts) | issuance + global + minting + transfer | ~15M | ~21M |
| Transfer (3 scripts) | base + global + transfer | ~12M | ~14M |
| Round-trip (4 scripts) | 2x base + global + transfer | ~14M | ~21M |

When the evaluator over-estimates beyond protocol limits, use manual exUnits
by removing the `evaluator` from `MeshTxBuilder` and passing `{ mem, steps }`
as the 3rd argument to redeemer methods.
